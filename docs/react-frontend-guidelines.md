# 前端开发规范

## 1. 目标

- 这份规范用于约束 `React + TypeScript + Vite` 前端开发。
- 目标不是只把页面做出来，而是保证页面结构清晰、交互稳定、可持续扩展。
- 通用工程规范继续复用后端已经沉淀的命名、注释、测试、重构、Review 标准。

## 2. 分层规范

- `pages/` 只负责页面级布局、路由入口、页面状态编排。
- `components/` 放可复用组件。
- `components/ui/` 只放基础 UI 组件，不放业务判断。
- `services/` 只负责请求接口、定义前端消费的 DTO，不写页面逻辑。
- `hooks/` 只抽复用状态和交互逻辑，不把页面主体塞进 hook。
- 一个页面里如果同时出现“路由控制、接口编排、复杂表单、多个业务块渲染”，必须拆组件或拆 hook。

## 3. 组件规范

- 页面组件优先保持薄，复杂块拆成子组件。
- 组件名必须表达业务语义，不允许 `Panel2`、`Box`、`DataView` 这种空名字。
- Props 只暴露当前组件真正需要的输入，不把整个对象随手往下传。
- 组件内部如果有关键交互策略，必须补解释性注释，说明为什么这样处理。
- 基础 UI 组件不允许直接依赖业务接口和业务数据结构。

## 4. Service 规范

- 所有接口请求统一收敛在 `services/`。
- 页面和组件不直接拼 URL，不直接写裸 `fetch`。
- Service 返回结构要对齐后端当前协议，不保留历史兼容壳。
- DTO 命名按接口语义命名，例如 `ContentDetail`、`ListedAsset`、`EditorDraft`。
- 一个接口协议收敛后，旧名字和旧字段要尽快清理，不长期并存。

## 5. 状态管理规范

- 页面私有状态优先用局部 `useState` / `useReducer`，不要过早引入全局状态。
- 异步状态至少明确区分：`loading`、`error`、`empty`、`ready`。
- 表单状态和服务端返回数据状态不要混在一个对象里随意覆盖。
- 能从现有状态推导出来的数据，用派生计算处理，不额外存一份副本。
- 如果一个状态值同时被三个以上区域依赖，先考虑抽 hook 或提升状态，而不是层层透传。

## 6. 表单与编辑器规范

- 编辑页必须明确区分“草稿保存”和“正式保存”。
- 表单字段名要和业务语义一致，不要用 `formData` 一把梭。
- 提交前先做最小必填校验，避免把明显错误直接打到后端。
- 有副作用的关键按钮必须有明确反馈，例如保存中、保存成功、保存失败。
- 编辑器涉及协议约束时，前端要遵守后端协议，不擅自发明新字段或新格式。

## 7. 交互状态规范

- 每个页面至少要有加载态、错误态、空态。
- 列表页在无数据时要显示明确空态，不允许只留白。
- 错误提示优先可理解，不直接把技术异常原样砸给用户。
- 异步请求期间，按钮和输入状态要避免产生重复提交和脏操作。
- 删除、覆盖、批量修改这类风险操作要有确认机制。

## 8. 视觉一致性规范

- 间距、边框、字号、色彩优先复用现有设计语言，不随页面自由发挥。
- 同类页面的结构骨架要一致，例如列表 + 详情、导航 + 工作区、表单 + 侧栏。
- 后台页优先强调信息密度和可操作性，不追求花哨装饰。
- 阅读页优先强调层级、留白和可读性，不用后台页样式硬套。
- 新页面如果明显偏离现有视觉语言，必须先确认方向再继续写。

## 9. 注释规范

- 关键实现必须写注释，说明这段交互或状态策略在解决什么问题。
- 注释重点写为什么这么做、在规避什么坑，不写翻译代码的废话。
- 简单映射、直接渲染、显然逻辑不强制注释。
- 代码改动导致注释失效时，必须同步更新。

## 10. 解释性注释要求

- 关键前端代码必须补解释性注释，不只交付“能跑”的实现。
- 注释至少要覆盖这三类信息：
  - 这段代码在解决什么问题。
  - 为什么采用当前策略，而不是更直觉的写法。
  - 这里在约束什么边界，或规避什么交互 / 状态问题。
- 以下场景默认必须写解释性注释：
  - 异步状态编排
  - 表单保存 / 草稿恢复
  - 页面级数据装载策略
  - 组件间状态同步
  - 协议约束落地
  - 不是唯一显然实现方案的逻辑
- 以下场景不要求机械加注释：
  - 直接渲染
  - 简单 props 透传
  - 一眼可读的映射和样式拼装
- 禁止写“给变量赋值”“调用接口获取数据”这类翻译式注释。
- 当实现策略变化时，旧注释必须同步更新，不允许注释和代码语义脱节。

## 11. 测试规范

- 复杂页面至少要覆盖关键业务流程测试。
- 组件测试优先测公开行为，不绑定实现细节。
- 修复交互 Bug 时必须补回归测试。
- 当前仓库还没有完整前端测试基线时，至少保证 `build` 和 `lint` 可过。

## 12. 直接判定为不合格的代码

- 组件里直接写裸 `fetch`。
- 基础 UI 组件里写业务逻辑。
- 为了图快保留旧接口壳和旧字段长期共存。
- 状态命名模糊，例如 `data`、`info`、`temp`。
- 一个页面文件塞满请求、树操作、表单、弹窗、渲染却没有拆分计划。
- 没有加载态、错误态、空态。
- 关键交互没有反馈，用户无法判断是否成功。

## 13. 当前项目额外约束

- 内容编辑必须围绕统一内容模型，不再区分旧的 `note/post` 独立接口层。
- Markdown 正文协议以后端为准，前端不引入分页语义到正文协议本身。
- DOC 节点必须绑定内容实体，前端创建流程也必须遵守这个约束。
- 资产路径按 `./assets/...` 使用，不自行拼出其他托管路径。
